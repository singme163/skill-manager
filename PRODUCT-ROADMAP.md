# Skill Manager 产品迭代规划（v2.2 → v3.0）

> 这是一份**产品视角**的前瞻规划，区别于仓库里已完成的 [ROADMAP.md]（那是 v1.1→v2.0 的技术迭代计划）。
> 建议落地时存为 `PRODUCT-ROADMAP.md`。

## Context

Skill Manager 已迭代到 v2.2，从"增删查小工具"长成一个成熟的 **skill 工作台**：管理（多工具/项目/插件）、装（五入口 + 发现页）、写好（Lint/编辑器/历史/模板）、看懂（离线翻译）、同步（git 多机 + 分享）。46 个测试、八个 Release、CI 全绿。

现在要回答的问题不再是"补哪个功能"，而是"**作为一款个人神器，靠什么持续拉开身位、值得被口口相传**"。

**定位（已确认）**：macOS 原生、离线优先、免费开源的个人 skill 工作台，服务同时使用多个 AI 编程工具的开发者/极客。以**体验深度**和**口碑传播**为导向，不追规模、不商业化。

**平台（已确认）**：专注做深 macOS，把系统能力（翻译、快捷指令、菜单栏、钥匙串、会话日志）吃透，不外扩。

**下一阶段三大战役（按优先级）**：① AI 辅助作者 → ② 发现与生态 → ③ 可观测与活跃度。

**反向清单（明确不做）**：云端后端、账号体系、基于遥测的推荐、跨平台 GUI 重写、任何付费墙。这些都与"个人神器 + 隐私优先"冲突，写下来是为了保持专注。

---

## 产品主线：从「管理」到「共创」与「洞察」

一句话叙事升级：**过去帮你"管好 skill"，接下来帮你"写出更好的 skill、找到更好的 skill、看清 skill 到底有没有用"。**

工作台的动词序列演进：
`发现 · 管理 · 写好 · 看懂 · 同步` → 新增 `共创（AI）· 洞察（可观测）`

三大战役正好对应这条主线上最痛的三个缺口。

---

## 战役一：AI 辅助作者（最高优先级）

**产品判断**：现在的「检查（Lint）」只会*指出*问题，用户还得自己改。而写好 skill 的最大杠杆——一段能可靠触发的 `description`——恰恰最难手写。把能力从"检查"升级到"生成/修复"，直接命中公众号文章里那句核心痛点"**写完不自动触发**"。这是与所有其他 skill 管理器拉开身位的地方：别人管文件，我帮你把 skill 写好。

**关键设计决策**：**自带 API Key（BYO Key）**。用户填自己的 Anthropic API Key，存系统钥匙串（复用 `GitHubAuth` 的钥匙串模式，`Sources/SkillManagerCore/GitHubAuth.swift`），调用直接走用户账号——既符合"数据不离开本机/不商业化"，又零运营成本。无 Key 时 AI 功能优雅隐藏，不影响其余功能。

### v2.3 —「Skill Doctor」：一键写好 description + 修复 Lint
- **描述生成/优化**：读 skill 正文 → 生成一段含触发关键词、说明使用场景的高质量 `description`，一键替换（复用编辑器的 `FrontmatterParser.settingKey`，`Sources/SkillManagerCore/FrontmatterParser.swift`）。
- **Lint 一键 AI 修复**：「检查」标签每条 findings 旁加「AI 修复」——name 与目录不一致、description 过短/缺触发提示、正文为空，都能一键改好（把 `SkillLinter.lint` 的结果作为 prompt 上下文，`Sources/SkillManagerCore/SkillLinter.swift`）。
- 设置页新增「AI（Anthropic API Key）」区，钥匙串存储，与现有 GitHub Token 区并列。
- 复用 v2.1/v2.2 的 `TranslationRunner` 异步交互范式做加载/结果回填。

### v2.4 —「AI 起草」：自然语言 → 完整 skill
- 新建 skill 时多一个入口：用一句话描述想要什么 → 生成完整 `SKILL.md`（frontmatter + 结构化正文 + 建议的 references/scripts 骨架），落到现有模板创建流程（`SkillInstaller.createTemplate`）。
- 正文润色/结构建议：对着编辑器里的 skill，给"这段可以更具体/缺少步骤/建议拆分"类建议。
- （探索）**触发力自评**：给 description + 几个示例任务，估计是否会命中，形成写作反馈闭环。

**这一战役的成功标志**：用户装了个英文烂描述的社区 skill，一键就能让它在自己的中文语境里可靠触发——"帮我把 skill 写好"成为被反复安利的记忆点。

## 战役二：发现与生态（第二优先级）

**产品判断**：发现页（v1.4）已经能浏览 + 一键装，但离"skill 生态入口"还差两步——**东西不够多**（收录门槛高）、**挑不出好的**（缺质量信号和分类）。个人神器要靠口碑传播，一个活跃的发现页是最好的传播飞轮：越多人来找 skill → 越多人 PR 收录自己的 skill → 越多人来找。

### v2.5 —「发现页 2.0」
- **分类 + 标签浏览 + 排序**：`index/skills.json` schema 加 category；发现页支持按分类筛选、按更新时间/热度排序（当前只有搜索）。
- **质量信号**：对每个条目拉取 GitHub stars / 最近更新时间（复用 `SkillInstaller` 的 GitHub 请求 + `GitHubAuth`），帮用户挑靠谱的。
- **一键提交收录**：详情页/我的 skill →「提交到发现页」，自动生成 JSON 条目 + 打开预填好的 GitHub PR（复用 `GitHubAuth`）。把社区贡献从"手写 JSON 提 PR"降到"点一下"——这是飞轮能否转起来的关键。
- 索引仓库加轻量校验 CI（JSON schema + 每个 url 可解析出 SKILL.md），保证收录质量、减轻人工审核。

**成功标志**：`index/skills.json` 的收录来源从"作者自己加"变成"社区 PR 涌入"，PR 数成为口碑的先行指标。

## 战役三：可观测与活跃度（第三优先级，探索性）

**产品判断**：这是最有想象力、也最有差异化的一战——**没有任何同类工具告诉你"装的 skill 到底有没有被触发过"**。Claude Code / Codex 都在本地留有会话日志（`~/.claude/projects/**`、`~/.codex/sessions/**`，环境里可见）。只读解析这些日志，就能把"盘点装了什么"升级为"洞察什么真在用"。全程本地、只读，契合隐私优先。

**风险前置**：日志格式未公开且会变，所以整战役框为**探索性**，先做一个能容错降级的最小版本，日志解析失败时功能静默隐藏、绝不影响主流程。

### v2.6 / v3.0 —「使用洞察」
- **触发统计**：解析会话日志，统计每个 skill 被加载/触发的次数与最近触发时间，显示在列表行与详情页（"本月触发 12 次 · 最近 3 天前"）。
- **僵尸 skill 提示**：90 天从未触发的 skill → 建议复查或删除，帮用户给臃肿的 skills 目录瘦身。
- **触发失败诊断**：你以为会触发却没触发的 skill → 联动 Lint，提示很可能是 description 触发词不足（顺势导流到战役一的"AI 修复"）。
- 所有统计本地计算、可一键关闭，绝不上传。

**成功标志**：用户第一次看到"这个装了半年的 skill 一次没触发过"时的惊讶——这种"只有这个工具能告诉我"的时刻，是最强的口碑燃料。

## 地基 track（你已降优先级，但诚实标注）

你没把"发布与信任"选进战役，我尊重这个判断。但从**口碑传播**目标看，它其实是当前最大的隐性阻碍，建议择机（尤其 Apple Developer 账号问题一旦解决时）顺手补上，不必单独占一个大版本：

- **签名公证**：目前 ad-hoc 签名，普通用户下载后被 Gatekeeper 拦、要"右键打开"——这一步会劝退相当比例的非技术传播对象。`Scripts/make-app.sh` / `make-dmg.sh` 的 `CODESIGN_IDENTITY` / `NOTARY_PROFILE` 钩子早已就位，有账号后一条命令即可产出公证 dmg。
- **Sparkle 自动更新**：迭代这么快，让老用户自动收到新版，是留存与口碑的杠杆。
- 判断：**签名公证 > Sparkle**。前者影响"能不能顺利装上"（决定第一印象），优先级实际上不低，建议插在战役一和战役二之间择机做掉。

## 成功度量（口碑导向）

主线指标（个人神器 + 口碑，不看营收/DAU）：
- **GitHub stars 增速** · **Release 下载数** · **issue/PR 活跃度**（社区愿不愿意参与）
- **发现页收录数与 PR 频率**（战役二飞轮是否转起来）
- **公众号文章 → 仓库的转化**（每次发版配一篇更新，形成内容-产品闭环）
- 战役三落地后，可增一个**产品内自证指标**："被洞察功能标记为僵尸并清理的 skill 数"——证明工具真的帮用户做了决策。

自用 dogfooding 是第一道验证：每个战役先在自己的 skills 上跑通、觉得"没它不行"，再发布。

## 风险与对策

| 风险 | 对策 |
|---|---|
| AI 功能依赖用户自备 Key，门槛高 | 明确定位为"进阶能力"，无 Key 时优雅隐藏；核心的管理/翻译/同步永远免费无门槛 |
| 会话日志格式未公开、会变（战役三） | 框为探索性，容错解析、失败静默降级，绝不阻塞主流程 |
| 发现页收录质量参差 | 索引仓库校验 CI + 一键 PR 里带 checklist，人工把关标准写进 index/README |
| Gatekeeper 摩擦劝退传播对象 | 地基 track 的签名公证择机补上；README/文章始终给"右键打开"引导兜底 |
| AI 生成内容质量不稳 | 生成结果永远进"编辑器待用户确认"，不自动落盘；配合现有编辑历史快照可回滚 |

---

## 验证方式（每个战役通用）

1. **产品验证（首要）**：作者本人 dogfooding——该战役的能力先在自己 27 个 skill 上跑一轮，达到"愿意主动安利"才算过。
2. **工程验证**：核心逻辑（description 生成的 prompt 组装、index schema 解析、日志解析器）补 Swift Testing 用例，跑 `Scripts/test.sh` 全绿；`Scripts/make-app.sh` 打包后真机冒烟主链路。
3. **界面验证**：用已有的 `Scripts/snapshots.sh` 截图流水线，为每个战役的新界面生成配图，顺带更新公众号文章形成内容闭环。
4. **发布验证**：打 tag → CI → Release dmg，与既有流程一致。
